## HTMLファイル

### 1. 命名ルールの破綻（l-header内にl-footer / serviceセクション内にp-medical）

**調査結果: 問題あり**

#### l-header / l-footer の構成

- `l-header` 内に `l-footer` が含まれる箇所は確認されませんでした。
- ヘッダーは `<header class="I-header">` → `I-header-wrapper` → `l-header__brand` + `l-header__actions` の構造です。
- フッターは `<footer class="I-footer">` として独立しています。
- ただし、ヘッダーのクラス名が `I-header`（大文字I）となっており、`l-header`（小文字L = Layout）の命名規則から外れている点があります。

#### serviceセクション内にp-medicalが存在

該当箇所: `index.html` 847〜896行目

```html
<div class="service-container-sp p-medical_voice_slider_wrapper">
  <button class="p-medical_voice_nav p-medical_voice_nav--prev">
  <button class="p-medical_voice_nav p-medical_voice_nav--next">
  <div class="p-medical_voice_grid">
    <a href="personal" class="p-medical_voice_card">
    <a href="sportip" class="p-medical_voice_card">
    <a href="about" class="p-medical_voice_card">
```

- トップページのサービスセクション内で `p-medical_voice_*` クラスが使用されています。
- `p-medical` はmedicalページ固有のプレフィクスですが、トップページのサービスセクションで使われているため命名規則が破綻しています。
- 本来は `c-service__card` や `p-top__service-card` のような命名が適切です。

**調査結果: 問題あり（全23ファイルで該当）**

全ページで以下のようなユーティリティクラスの組み合わせが多用されています。

### 3. PCとSPで2ソースに分割されているコード

**調査結果: 問題あり**

#### c-shop セクション（`index.html` 1055〜1159行目, `about/index.html` 600〜645行目）

PC用とSP用でHTML構造を完全に複製しています。

```html
<!-- PCヘッダー -->
<div class="reveal c-shop__header--desk">...</div>

<!-- SPヘッダー -->
<div class="c-shop__header--sp">...</div>

<!-- SPマップボックス -->
<figure class="c-shop__map-box c-shop__map-box--sp">...</figure>

<!-- PCマップボックス -->
<figure class="c-shop__map-box">...</figure>

<!-- SPステッカー -->
<aside class="c-shop__sticker-sp c-shop__sticker-container relative">...</aside>

<!-- PCステッカー -->
<aside class="c-shop__sticker-desk">...</aside>
```

- 同じコンテンツが2つのHTML構造で記述され、CSSの `display` で表示/非表示を切り替えています。
- レスポンシブ対応はCSSのみで制御するのが望ましく、HTML構造の複製は保守性を下げます。

#### l-header__wrapper

- 現状は2ソース分割ではなく、単一構造でCSSメディアクエリで対応しています（問題なし）。

### 5. ユーティリティクラスの命名と役割の乖離

**調査結果: 問題あり**

お客様ご指摘の `.p-top-hero__text .u-fs19{ font-size: 15px; }` のようなパターンは、現在のコードでは `u-` プレフィクスのユーティリティクラスは使用されておらず、代わりに `fs-14`, `fw-500`, `text-white` のような短縮形のユーティリティクラスに変更されています。

ただし、同様の問題として:
- ユーティリティクラス（`fs-14` = font-size 14px）をコンポーネントクラスと組み合わせて使い、CSSのメディアクエリでサイズを上書きしているケースがあります。
- これにより「`fs-14` なのに実際は別のサイズ」という状態が発生する可能性があります。

## JSファイル

### 8. global.js と common.js の分割理由

**調査結果: 分割自体には理由あり。ただし問題点あり**

#### 各JSファイルの行数と役割

| ファイル | 行数 | 役割 |
|---|---|---|
| global.js | 1,088行 | ページ全体のグローバル機能（スクロール制御、メニュー、スライダー、タブ、アニメーション等） |
| common.js | 100行 | 再利用可能なユーティリティ関数（`initSmoothScroll`, `initMobileCarousel`, `initModal`） |
| public-main.js | 1,788行 | Google Reviewsプラグイン用（WordPress由来、削除可能） |
| storedetail.js | 118行 | 店舗詳細ページ用（スムーススクロール、ギャラリー、スライダー） |
| top.js | 105行 | トップページ用（ヒーロースライダー制御） |
| family.js | 27行 | ファミリーページ用（FAQ制御） |
| protein.js | 20行 | プロテインページ用（モーダル、カルーセル） |
| medical.js | 7行 | 医療ページ用 |
| payment.js | 1行 | 決済ページ用（`initSmoothScroll` 呼出のみ） |
| reservation.js | 1行 | 予約ページ用（`initSmoothScroll` 呼出のみ） |
| price.js | 1行 | 価格ページ用（`initSmoothScroll` 呼出のみ） |
| sportip.js | 1行 | スポーティップページ用（`initSmoothScroll` 呼出のみ） |

#### 分割の意図

- **common.js**: モジュール化された再利用関数（`initSmoothScroll`, `initMobileCarousel`, `initModal`）
  - payment.js, reservation.js, price.js, sportip.js, medical.js, protein.js が依存
- **global.js**: ページ全体の状態管理・初期化（スクロール監視、メニュー、アニメーション等）

分割自体は「再利用可能な関数」と「グローバル初期化」を分離する意図として理解できます。

#### 問題点

1. **`init__tabs()` 関数が global.js 内で2回定義されている**（833行目: 空関数 / 841行目: 実装版）
2. **スクロールアップが2箇所で重複実装**
   - global.js 2〜7行目: `smoothScrollTo(0, 600)` カスタム関数版
   - global.js 1087〜1088行目: `window.scrollTo({ top: 0, behavior: "smooth" })` ネイティブ版

### 9. スムーズスクロールの処理が複数点在

**調査結果: 問題あり（3種類の実装が混在、計12箇所）**

| ファイル | 行 | 実装方法 | 対象セレクタ |
|---|---|---|---|
| global.js | 5 | `smoothScrollTo()` カスタム関数 | `.scroll-up` |
| global.js | 32 | `smoothScrollTo()` カスタム関数 | セクション自動スクロール |
| global.js | 763 | `scrollIntoView({ behavior: 'smooth' })` | 全ハッシュアンカー `a[href^="#"]` |
| global.js | 905 | `container.scrollTo()` | タブボタン横スクロール |
| global.js | 914 | `window.scrollTo()` | タブターゲットへのスクロール |
| global.js | 1088 | `window.scrollTo()` | スクロールアップボタン |
| common.js | 18 | `scrollIntoView({ behavior: 'smooth' })` | `initSmoothScroll()` 関数 |
| storedetail.js | 11 | `scrollIntoView({ behavior: 'smooth' })` | `.p-storedetail_links-nav a` |
| payment.js | 1 | `initSmoothScroll()` (common.js経由) | `.p-payment__links-nav a` |
| reservation.js | 1 | `initSmoothScroll()` (common.js経由) | `.p-reservation-flow__nav a` |
| price.js | 1 | `initSmoothScroll()` (common.js経由) | `.p-price__links-nav a` |
| sportip.js | 1 | `initSmoothScroll()` (common.js経由) | `.p-sportip__links-nav a` |

#### 混在する実装方法

1. **カスタム関数 `smoothScrollTo()`** — `requestAnimationFrame` ベース（global.js独自）
2. **`scrollIntoView({ behavior: 'smooth' })`** — ブラウザネイティブ（common.js / storedetail.js）
3. **`window.scrollTo({ behavior: 'smooth' })`** — ブラウザネイティブ別パターン（global.js内）

#### その他の重複

**モバイルカルーセルの重複実装:**
- common.js の `initMobileCarousel()` と storedetail.js の独自カルーセル（93〜118行目）がほぼ同じ処理

**モバイル判定のブレークポイント不統一:**

| ファイル | ブレークポイント |
|---|---|
| common.js | `<= 767px` |
| family.js | `<= 767px` |
| storedetail.js | `<= 1024px` |
| global.js | `<= 1091px` |

- 同じ「モバイル判定」の目的で4つの異なるブレークポイントが定義されています。
