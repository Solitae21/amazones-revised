## プロジェクト概要

Amazones（アマゾネス）ブランディングサイトのコーディング品質改善プロジェクト。
静的HTML/CSS/JSサイト（PHP対応可）。WordPressからの移行済み。

---

## 最重要原則

1. **コーディングガイドライン（`docs/cording-guideline.md`）準拠を最優先する** — ガイドラインからの逸脱を発見した場合は、スコープ拡大してでも是正対象とする。既存コードがガイドラインから外れている場合でも「既存パターンだから」を理由に踏襲しない。ガイドラインと既存コードが矛盾する場合は**ガイドラインが「正」**。
2. **見た目を絶対に崩さない** — リファクタリングはすべて内部構造の改善。ブラウザ上の表示に影響を与えない。
3. **先方に価値を届ける速度を最優先する** — Plan 作成・文書化より実作業を進める。数字の改善と質的改善を混同しない。ただし原則 1・2 に劣後する。
4. **判断に迷う場合は修正しない**（安全側に倒す）。人間に確認する。

以下のタイミングで「年収3000万円のフロントエンドエンジニアとして、客観的な視点で、先方フィードバックに記述されている根本原因が解消された内容になっていますか」と聞かれたら、該当する観点で評価すること。

### Plan 作成後

- その Plan は先方フィードバック（`docs/plans/feedback-improvement-context.md`）の根本原因を解決するか
- 作業の優先順位は「先方に価値を届ける速度」の観点で正しいか
- Plan 作成自体に工数をかけすぎていないか
- 先方確認のタイミングは適切か（手戻りリスクの最小化）

### 修正完了後

- 先方フィードバックの根本原因が解消された内容になっているか
- 先方が「改善された」と実感できる質的変化があるか（数字だけの改善になっていないか）
- HTML の可読性・保守性は具体的にどう改善されたか（Before/After で説明できるか）
- 見た目を崩すリスクはないか
- 次に先方に共有できる状態になっているか

## Phase 2-B: ページ固有ユーティリティ改善の進め方

Phase 2-B は残り22ページに対して、index.html（Phase 2-A）と同じ作業を展開する。
担当割り振り・人間向け手順は `docs/phase2b-guide.md` を参照。

### Plan 作成のコツ

1. **まず対象ページの HTML を読み、SCSS を読む** — Plan を書く前に現状を把握する
2. **実際のファイルの行番号・コードを確認してから Plan に書く** — コンテキストファイルや過去の Plan のコードをそのまま流用しない
3. **1チケット = 1機能に絞る** — スコープが大きいと整合性が崩れる。Plan は必ずチケット単位に分割し、1つずつ渡す
4. **チケット = 「何を・なぜ・受け入れ基準」のみ** — 具体的な SCSS コードは Plan に書かない。「どう実装するか」は既存コード + ガイドラインから判断する（Plan を「正」として扱わない）
5. **影響ファイルを明記する** — そのページだけか、他ページにも影響するか
6. **検証方法を必ず含める** — grep コマンド、コンパイル確認、ブラウザ確認の3点セット
7. **お手本**: `docs/plans/feedback-index-plan.md` の Phase 2 セクションと `docs/plans/utility-cleanup-plan.md`

### Plan 作成後の自動フロー

Plan を作成したら、以下を自動で行うこと:

1. **セルフレビュー**（ユーザーには出力しない）
   - 「年収3000万円のフロントエンドエンジニアとして、先方フィードバック（`docs/plans/feedback-improvement-context.md`）の根本原因が解消された内容になっているか」を確認
   - レビュー観点は「レビュープロンプト > Plan 作成後」セクションに準拠
   - 問題があれば Plan を修正する
2. **Plan の概要（チケット一覧表）を出力**し、ユーザーの承認を待つ
3. ユーザーが承認したら、**チケットごとの実行プロンプト**（対象ファイル・行番号・作業内容・検証コマンド含む）を出力する
4. 最後に「**トークン節約のため、`/clear` コマンドでこのチャットを終了して、新規チャットに上記の実行プロンプトを入力してください。**」と伝える

### 作業実行のコツ

1. **タイポグラフィ系 u-\* を SCSS に吸収する**（u-fs-\*, u-fw-\*, u-text-\*, u-lh-\*, u-font-\*, u-ls-\*）
2. **レイアウト・機能系 u-\* は残す**（u-relative, u-display-sp, u-mx-auto, u-mt-\* 等）
3. **body 継承で不要な u-\* は SCSS 追記なしで削除するだけ**（u-fw-500, u-fs-16, u-font-zen, u-text-black1）
4. **吸収パターンは3つ**:
   - 既存クラスに SCSS 追記 → HTML から u-\* 削除
   - クラスがない要素 → BEM 子要素クラスを新設
   - body 継承で不要 → 削除のみ
5. **スペーシング系 u-\*（u-mt-\*, u-mb-\*, u-py-\* 等）は既存クラスに吸収する。吸収先がなければ u-\* のまま残す** — margin/padding の単一プロパティだけを持つ新クラスを作ってはいけない（u-\* の名前を p-\* に変えただけの焼き直し）
6. **セレクタ深さは2階層まで**（ガイドライン準拠）
7. **カラーは変数を使う**（`var(--text-white)` 等）
8. **他ページの HTML・SCSS は変更しない**（自分の担当ページのみ）
9. **共通コンポーネント（ヘッダー、フッター等）は触らない**（Phase 1 で対応済み）

## SCSS構成

```
assets/scss/
├── global.scss              # エントリポイント
├── _global-responsive.scss  # グローバルのレスポンシブ
├── foundation/
│   ├── _variables.scss      # カラー・フォントサイズ変数
│   └── _reset.scss          # リセットCSS
├── layout/
│   ├── _header.scss         # ヘッダー
│   └── _footer.scss         # フッター
├── object/
│   ├── component/           # 再利用コンポーネント (c-*)
│   ├── project/             # ページ固有スタイル (p-*)
│   └── utility/             # ユーティリティ (u-*)
```
